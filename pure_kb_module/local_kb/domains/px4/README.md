# PX4 领域入口

## 这份文档负责什么

这份文档负责：

- 作为 `PX4` 领域的总入口
- 告诉 AI 命中 `PX4` 后应先如何进入本领域
- 说明纯模块阶段 `PX4` 领域的可移植边界和后续挂接方式

这份文档不负责：

- 伪装成已经自带完整 `PX4` 私有知识库
- 直接绑定旧工作区中的私有子知识包路径
- 代替未来真实 `PX4` 正式条目承载具体个案细节

## 语义位置

推荐在知识图谱中把这里理解为：

- `工程知识`
- `自动控制`
- `PX4`

而不是一个孤立目录。

当前纯模块中只保留 `PX4` 领域入口，不附带私有子知识包。

如果后续要在新工程里接入具体 `PX4` 子知识库，建议：

1. 保持总库先命中 `PX4` 领域
2. 再按新工程自己的 `kb_manifest` 接入对应子知识包
3. 不要在纯模块里直接绑定旧工作区中的私有路径

## 当前边界

- 当前包含：
  - `PX4` 领域入口
  - 进入 `PX4` 语义后的最小规则
  - 后续挂接真实 `PX4` 子知识库的方式说明
- 当前不包含：
  - 原工作区私有 `PX4` 子知识包
  - 真实工程个案知识
  - 长链路的具体排查条目

## 当前纯模块阶段的 PX4 入口规则

**结构化元信息**

- `layer`: `entrypoint`
- `symptom`: 当前请求已经命中 `PX4`，但纯模块里还没有接入具体 `PX4` 个案知识，导致 AI 容易退回到通用治理文档
- `evidence`:
  - 纯模块只保留领域入口，不附带私有 `PX4` 子知识包
  - 新工程测试阶段更需要先验证总路由、领域入口和后续挂接方式
- `recommended_action`:
  - 命中 `PX4` 后先读取本页作为领域入口
  - 如果新工程已有自己的 `PX4` 知识包，再通过该工程的 `manifest` 继续挂接
  - 如果当前工程还没有 `PX4` 知识内容，先保留为领域入口而不是误读旧工作区子库
- `portability`: `high`
- `confidence`: `verified`

## AI 进入本领域后的最小动作

1. 总库先判定领域为 `PX4`
2. 先读取本页作为领域入口
3. 如果当前工程已有自己的 `PX4` 知识包，再继续读取该工程最相关的 `PX4` 子知识
4. 如果当前工程还没有 `PX4` 正式知识，先停在领域入口，不误读旧工作区私有路径
